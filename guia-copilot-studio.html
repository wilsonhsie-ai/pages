Criação da Conta PAYG (Pay-As-You-Go)

O que é o modelo PAYG?

O modelo Pay-As-You-Go permite usar o Copilot Studio sem pacotes pré-pagos. A cobrança é feita com base no consumo real de mensagens/créditos, vinculada diretamente a uma assinatura Azure. É ideal para pilotos e início de jornada, pois não exige compromisso financeiro antecipado.

💡 Importante

A partir de setembro de 2025, o Copilot Studio migrou de "mensagens" para "Copilot Credits", onde cada tipo de interação consome diferentes quantidades de créditos (respostas básicas, ações generativas com IA, chamadas de API, etc.).

Pré-requisitos

Requisito Detalhes
Assinatura Azure Ativa, com permissão de Owner ou Contributor no Resource Group
Resource Group Criado dentro da assinatura Azure para rastrear custos
Ambiente Power Platform Com Dataverse habilitado (Sandbox ou Production)
Permissões de Admin Power Platform Admin, Global Admin ou Dynamics 365 Admin
Tenant vinculado Mesmo tenant do Microsoft 365 e do Azure

Passo a Passo de Configuração

  1. Criar Assinatura e Resource Group no Azure
    Acesse o Portal Azure → Subscriptions → Adicionar (se necessário). Em seguida, crie um Resource Group dedicado (ex: RG-CopilotStudio) para rastrear custos separadamente.
  2. Acessar o Power Platform Admin Center
    Navegue até admin.powerplatform.microsoft.com com credenciais de administrador.
  3. Criar Billing Policy (Plano de Faturamento)
    No menu lateral, vá em Billing → Billing Plans → New Billing Plan. Selecione a assinatura Azure e o Resource Group criado no passo 1. Escolha Copilot Studio como produto habilitado.
  4. Vincular o Ambiente ao Billing Policy
    Associe o Billing Policy ao ambiente Dataverse desejado. Todo consumo desse ambiente será medido e cobrado via Azure.
  5. Verificar no Azure Cost Management
    Após a vinculação, acesse Azure Portal → Cost Management + Billing para confirmar que o recurso está aparecendo e configurar alertas de orçamento.
  6. Configurar Alertas de Custo
    Em Azure Cost Management, crie Budget Alerts (ex: R$ 500, R$ 1.000, R$ 5.000) para receber notificações antes de atingir limites.
⚠️ Recomendação

Crie um Resource Group dedicado por cada ambiente (Dev e Prod) para facilitar o controle de custos. Exemplo: RG-CopilotStudio-Dev e RG-CopilotStudio-Prod.

PAYG vs. Pacotes Pré-pagos

Aspecto PAYG Pacotes Pré-pagos
Compromisso inicial Nenhum Valor fixo
Ideal para Pilotos, validação, baixo volume Produção estabilizada, previsibilidade
Controle de custo Alertas no Azure Cost Management Limite natural do pacote
Flexibilidade Alta Moderada
Combinação ✅ É possível combinar PAYG + Pacotes (base pré-paga com overflow via PAYG)

Atribuição de Usuários Criadores de Agentes

Quem pode criar agentes?

Para que um usuário consiga criar e editar agentes no Copilot Studio, ele precisa ter o papel de Environment Maker (ou um papel customizado equivalente) no ambiente Power Platform onde vai trabalhar.

Método Recomendado: Grupos de Segurança via Entra ID

Em vez de atribuir permissões individualmente, utilize Security Groups do Microsoft Entra ID para gerenciar os criadores de forma escalável:

  1. Criar Security Group no Microsoft Entra
    Acesse o Microsoft Entra Admin Center → Groups → New Group. Nomeie como SG-AgentMakers. Adicione os usuários que serão criadores de agentes.
  2. Associar o Grupo ao Ambiente no Power Platform
    No Power Platform Admin Center → Environments → selecione o ambiente → Users + Permissions → Teams → Create Team. Escolha o tipo Microsoft Entra ID Security Group e selecione o grupo criado.
  3. Atribuir o Papel "Environment Maker"
    Com o team criado, atribua o papel Environment Maker. Todos os membros do grupo herdarão as permissões automaticamente.
  4. Validar o Acesso
    Peça a um membro do grupo acessar copilotstudio.microsoft.com e verificar se consegue criar um novo agente no ambiente correto.
✅ Benefício do Grupo de Segurança

Quando um novo colaborador é adicionado ao grupo no Entra, ele recebe permissões automaticamente no Copilot Studio — sem intervenção manual. Da mesma forma, ao remover, o acesso é revogado instantaneamente.

Roles Customizados (para controle granular)

Se a organização quiser dar acesso ao Copilot Studio sem conceder todos os privilégios do Environment Maker, é possível criar Custom Dataverse Security Roles:

  • Permissão de leitura/escrita apenas nas tabelas do Copilot Studio
  • Bloquear criação de Power Apps, Power Automate flows, etc.
  • Seguir o princípio do menor privilégio
  • Atribuir o role customizado ao Team/Security Group

Sugestão de Grupos

Grupo Entra ID Papel Ambiente Descrição
SG-AgentMakers-Dev Environment Maker Desenvolvimento Criadores com liberdade para experimentar
SG-AgentMakers-Prod Custom Role (restrito) Produção Apenas deploy, sem alteração livre
SG-AgentAdmins System Admin Todos Administradores de governança e billing

Procedimento de Publicação

  1. Desenvolver e Testar o Agente
    Crie o agente no ambiente de Desenvolvimento. Utilize o Test Chat integrado do Copilot Studio (que não consome mensagens faturáveis) para validar tópicos, fontes de conhecimento e ações.
  2. Publicar no Ambiente de Teste
    No Copilot Studio, clique em Publish. Configure os canais de teste (ex: Teams com um grupo piloto de 5-10 usuários) para validação com dados reais.
  3. Aprovar via Workflow (Opcional mas recomendado)
    Configure um Power Automate Approval Flow para que toda publicação em Produção passe por aprovação do responsável da área ou do TI. Isso garante que apenas agentes revisados cheguem aos usuários finais.
  4. Exportar como Solução (ALM)
    Para promover o agente do ambiente Dev para Prod, exporte-o como uma Managed Solution via Power Platform → Solutions. Isso garante rastreabilidade e controle de versão.
  5. Importar no Ambiente de Produção
    No ambiente de Produção, importe a Managed Solution. Configure os canais definitivos (Microsoft Teams, SharePoint, website, etc.).
  6. Ativar Canais de Distribuição
    Configure os canais onde o agente ficará disponível para os usuários finais.

Canais de Publicação Disponíveis

Canal Uso Recomendado Observações
Microsoft Teams Agentes internos (RH, TI, Suprimentos) Mais rápido de configurar, autenticação SSO nativa
SharePoint FAQ da intranet, bases de conhecimento Integração com páginas existentes
Website / iFrame Atendimento a clientes, fornecedores Requer configuração de autenticação
Mobile App Field service, força de vendas Via Power Apps ou embed
💡 Dica: Application Lifecycle Management (ALM)

Utilize Solutions do Power Platform para gerenciar o ciclo de vida dos agentes. Exporte como Unmanaged do Dev e importe como Managed no Prod. Isso impede alterações acidentais em Produção e permite rollback se necessário.

Onde Monitorar

Ferramenta O que Mostra Nível de Detalhe

Copilot Studio Analytics Volume de conversas, performance de tópicos, satisfação do usuário, tendências Alto

Mensagens Faturáveis vs. Não Faturáveis

Tipo Exemplos Cobrado?
Faturável Interação de usuário final que gera resposta do agente, ações generativas com IA, chamadas de API Sim
Não Faturável Teste interno no Test Chat do Copilot Studio, chamadas de sistema, retries Não

Configuração do Azure Application Insights (Recomendado)

  1. Criar recurso Application Insights no Azure
    Azure Portal → Create Resource → Application Insights. Nomeie como AI-CopilotStudio.
  2. Copiar a Connection String
    Na página do recurso, copie a Connection String (não mais Instrumentation Key).
  3. Vincular ao Agente no Copilot Studio
    No Copilot Studio → Settings do agente → Advanced → cole a Connection String do Application Insights.
  4. Consultar via KQL
    Use queries KQL no Application Insights para criar dashboards customizados de consumo, tempo de resposta e erros.

🧮 Calculadora de Estimativa de Copilot Credits

Antes de publicar um agente em produção, é fundamental estimar o consumo de Copilot Credits para dimensionar corretamente seu Billing Policy e evitar surpresas no custo. A Microsoft disponibiliza uma calculadora oficial para isso:

🔗 Copilot Studio Estimator

https://microsoft.github.io/copilot-studio-estimator/

Ferramenta interativa que calcula a estimativa de Copilot Credits com base no perfil do seu agente. Não é uma oferta vinculante — use como referência para planejamento.

Como usar a calculadora:

  1. Selecione a categoria do agente
    Escolha uma ou mais categorias (ex: atendimento ao cliente, RH, TI) para filtrar os tipos de agente disponíveis.
  2. Configure o tráfego do agente
    Informe: quantos usuários acessarão o agente e quantas interações por mês cada usuário fará em média. Indique também quantos possuem licença M365 Copilot (esses têm créditos inclusos).
  3. Defina as fontes de conhecimento
    Qual percentual de respostas usa knowledge? Dessas, quantas usam Tenant Graph Grounding (SharePoint, emails, calendário) vs. outras fontes (Dataverse, websites, arquivos)?
  4. Configure as ferramentas (Tools)
    Adicione os tipos de tool que o agente usa: Prompts, Agent Flows, Custom Connectors, REST APIs, MCP. Informe quantidade e frequência de invocação por interação.
  5. Revise os modificadores opcionais
    Se usar nós de Prompt (Basic, Standard ou Premium), informe a quantidade e frequência de disparo.

O que a calculadora retorna:

Componente O que Calcula
Total de Copilot Credits Estimativa total mensal de créditos consumidos pelo agente
Credits por Knowledge Créditos para respostas via Tenant Graph Grounding e outras fontes de conhecimento
Credits por Agent Flows Cobrados a 13 créditos por 100 ações (0,13/ação) — custo fixo independente de usuários
Credits por Modificadores Créditos adicionais para nós de Prompt (Basic, Standard, Premium)
Negação M365 Copilot Créditos descontados para usuários com licença M365 Copilot (já incluem allocation)
⚠️ Importante sobre a Estimativa

A calculadora é uma ferramenta de referência, não uma oferta vinculante. O consumo real pode variar conforme região, workload, número de usuários e complexidade das interações. Use-a para dimensionamento inicial e depois monitore o consumo real com as ferramentas da seção acima.

✅ Monitoramento Avançado com Purview

Para requisitos de compliance e auditoria, o Microsoft Purview pode monitorar eventos de ciclo de vida dos agentes, acesso a dados sensíveis e comunicações — ideal para empresas com regulamentações setoriais e LGPD.

Governança de Agentes — Melhores Práticas

Princípio Fundamental: Guardrails, não Gates

A melhor prática da Microsoft é empoderar a inovação com proteção, em vez de bloquear completamente a criação de agentes. Crie barreiras inteligentes que orientem os usuários a criar agentes de qualidade, sem sufocar a inovação.

Pilares de Governança

1. DLP (Data Loss Prevention)

Configure políticas de DLP no nível do ambiente para controlar quais conectores os agentes podem utilizar. Exemplo: permitir SharePoint e Dataverse, mas bloquear conectores de redes sociais ou armazenamento externo.

2. Managed Environments

Habilite Managed Environments para cada ambiente Power Platform. Isso permite:

  • Limitar quem pode compartilhar apps e agentes
  • Bloquear publicação sem aprovação
  • Requerer compliance com políticas de dados
  • Monitorar ativamente os recursos criados

3. RACI Matrix para Agentes

Atividade Responsável (R) Aprovador (A) Consultado (C) Informado (I)
Criação de agente Maker da área — TI / CoE Gestor da área
Teste e validação Maker + usuários piloto Gestor da área TI —
Publicação em Prod TI / CoE Gestor TI Maker Compliance
Monitoramento contínuo TI / CoE — — Gestor da área

4. Center of Excellence (CoE) Starter Kit

Implemente o CoE Starter Kit da Microsoft para ter visibilidade total de:

  • Quantos agentes existem no tenant
  • Quem criou cada agente e quando
  • Quais conectores estão sendo usados
  • Agentes órfãos (sem dono ativo)
  • Compliance com políticas de DLP

Zonas de Governança (Traffic Light Model)

🟢 Zona Verde

Sandbox / Experimentação
DLP permissivo. Makers podem testar livremente. Sem impacto em produção.

🟡 Zona Amarela

Desenvolvimento Controlado
DLP moderado. Conectores aprovados apenas. Review obrigatório antes de promover.

🔴 Zona Vermelha

Produção
DLP restritivo. Apenas Managed Solutions. Auditoria completa. Compliance obrigatório.

Estratégia de Ambientes: Dev vs. Prod vs. Por Departamento

As Duas Abordagens

✅ Recomendado: Dois Ambientes (Dev + Prod)
  • Simplicidade operacional — menos overhead de gestão
  • Rateio de custos pelo Copilot Studio Analytics (por agente)
  • DLP e governança consistentes em toda a organização
  • ALM centralizado (Solutions) para promoção de agentes
  • Ideal para o início da jornada (até ~20 agentes)
  • Custo menor de Dataverse (apenas 2 ambientes)
Alternativa: Ambiente por Departamento
  • Isolamento de dados entre departamentos (Comercial, RH, TI, etc.)
  • Rateio de custos nativo por ambiente/Resource Group
  • DLP customizado por departamento
  • Complexidade alta de gestão (múltiplos ambientes × stages)
  • Custo maior de Dataverse (cada ambiente = capacidade separada)
  • Recomendado apenas em cenários regulatórios ou com +50 agentes
🎯 Recomendação

Para organizações no início da jornada, recomendamos fortemente a abordagem Dois Ambientes (Dev + Prod). Isso minimiza complexidade operacional e custos, enquanto o rateio entre departamentos pode ser feito pelas ferramentas de analytics do Copilot Studio e Azure Cost Management por agente/tag.

Quando a maturidade crescer (acima de 20-30 agentes, múltiplas áreas construindo de forma independente), poderá evoluir para ambientes departamentais de forma gradual.

Cotas, Rateio e Controle de Custos

É possível criar cotas por área / agente / projeto?

💡 Resposta Direta

O Copilot Studio não oferece nativamente cotas granulares por agente individual ou por departamento dentro de um mesmo ambiente. Porém, há estratégias eficazes para alcançar o mesmo resultado:

Estratégias de Rateio e Controle

Estratégia Como funciona Nível de Granularidade
1. Billing Policies por Ambiente Cada ambiente pode ter seu próprio Billing Policy vinculado a um Resource Group Azure diferente. Custos separados automaticamente. Por ambiente
2. Tags no Azure Adicione tags de departamento/projeto ao Resource Group (ex: dept=RH, projeto=Onboarding). Filtre custos no Azure Cost Management. Por projeto/dept
3. Analytics do Copilot Studio O painel de analytics mostra consumo por agente individual. Exporte dados para relatórios de rateio interno. Por agente
4. Budget Alerts no Azure Configure alertas de custo por Resource Group. Não impede o consumo, mas alerta quando limiares são atingidos. Por ambiente
5. Múltiplos Billing Policies Até 50 Billing Policies por tenant. Cada um pode ser vinculado a Resource Groups diferentes para isolamento financeiro. Por ambiente

Limites e Quotas do Copilot Studio

Recurso Limite
Requests por minuto (por ambiente, pago) 8.000 req/min
Requests por 24h (plano standard) 250.000
Requests por dia (plano Teams) 6.000
Tópicos por agente (Web App) 1.000
Tópicos por agente (Teams) 250
Billing Policies por tenant 50

Modelo de Rateio Sugerido

Fase 1 (Início): Um único Billing Policy PAYG → monitorar consumo por agente via analytics → criar relatório mensal de rateio manual.


Fase 2 (Escala): Se necessário, criar Billing Policies separados por área → cada um vinculado a Resource Groups com tags → automatizar relatórios via Power BI.


Fase 3 (Maturidade): Considerar ambientes separados por departamento + pacotes pré-pagos para áreas com consumo previsível.

Conectores do Power Platform

💡 O que são Conectores?

Conectores são pontes pré-construídas que permitem aos agentes do Copilot Studio se comunicar com serviços externos — como SharePoint, Outlook, SAP, Salesforce, SQL Server, APIs REST e centenas de outros. Eles transformam agentes conversacionais em agentes que executam ações reais: consultar dados, criar registros, enviar notificações e automatizar processos.

Como Adicionar Conectores a um Agente

  1. Acesse o Copilot Studio
    Abra seu agente em copilotstudio.microsoft.com e navegue até Ações (Actions) no menu lateral.
  2. Adicionar uma Ação
    Clique em "+ Adicionar uma ação". Você verá opções como: Conectores do Power Platform, Power Automate Flows, HTTP Request, API Plugin e Azure AI Foundry.
  3. Escolher o Conector
    Pesquise pelo nome do serviço (ex: "SharePoint", "Outlook", "SQL Server"). Selecione a operação desejada (ex: "Criar item em lista", "Enviar email", "Executar query").
  4. Configurar Autenticação
    Dependendo do conector, será necessário configurar a autenticação: OAuth 2.0, API Key ou credenciais de serviço. Conectores Microsoft geralmente usam o login do usuário (delegated auth).
  5. Mapear Parâmetros
    Vincule os inputs da ação aos dados extraídos da conversa. O Copilot Studio permite mapear variáveis do tópico diretamente aos parâmetros do conector.

Conectores Padrão vs. Premium

⚠️ Atenção ao Tipo de Conector

O tipo do conector (Standard ou Premium) impacta diretamente o custo e o licenciamento necessário. Identifique o tipo antes de usar em produção.

Característica Standard (Padrão) Premium
Custo adicional Não — incluso no Copilot Studio Sim — requer licença Premium ou PAYG
Identificação Sem ícone especial no catálogo Ícone de diamante 💎 ou label "Premium" no catálogo
Exemplos SharePoint, Outlook, Office 365, OneDrive, Teams, Excel Online, Approvals, Notifications SQL Server, Dataverse, Azure SQL, SAP, Salesforce, ServiceNow, HTTP (custom), Azure Blob, Power BI

Como Identificar se um Conector é Premium

  1. No Copilot Studio
    Ao adicionar uma ação, conectores premium exibem o ícone de diamante 💎 ao lado do nome ou a label "Premium" abaixo da descrição.
  2. Na Documentação
    Acesse a Referência de Conectores no Microsoft Learn. Cada conector lista seu tipo (Standard / Premium) na página de detalhes.
  3. No Power Platform Admin Center
    Em Políticas DLP, ao classificar conectores, o tipo aparece ao lado de cada um, facilitando a governança.

Boas Práticas para Conectores

✅ Recomendações
  • Comece com conectores Standard — valide o caso de uso antes de adotar conectores Premium
  • Classifique conectores nas políticas DLP — defina quais podem ser usados em cada ambiente (Business / Non-Business / Blocked)
  • Use Service Principals para conectores Premium em produção — evite dependência de conta de usuário
  • Monitore o consumo — cada execução de ação consome Copilot Credits; acompanhe via Power Platform Admin Center
  • Prefira Power Automate Flows para lógica complexa com múltiplos conectores — o agente chama um único flow que orquestra várias ações
  • Documente as conexões — mantenha um registro de quais agentes usam quais conectores para facilitar auditoria e troubleshooting

Arquitetura: Base de Conhecimento com Azure AI Foundry

O Desafio: Limitações do SharePoint Nativo

Agentes do Copilot Studio que consomem documentos diretamente do SharePoint enfrentam limitações importantes que podem comprometer a qualidade das respostas:

⚠️ Limitações do Conector SharePoint Nativo
  • Limite prático de 3–7 MB por arquivo — documentos maiores são parcialmente ignorados
  • PDFs escaneados (imagem) possuem OCR básico — conteúdo frequentemente perdido
  • Documentos com tabelas complexas e imagens perdem contexto na extração
  • Conteúdo útil por consulta limitado a ~36.000 caracteres (~15–20 páginas)
  • Indexação do SharePoint pode levar 4–6 horas após upload

Opções de Arquitetura Comparadas

Abordagem Performance Custo Limite/Arquivo Melhor Para
① SharePoint Direto ⭐⭐ Incluso M365 3–7 MB <1.000 docs simples e pequenos
② Upload Direto + Dataverse ⭐⭐⭐ Power Platform 512 MB <20 arquivos focados
③ Power Automate → Dataverse ⭐⭐⭐ Power Platform 512 MB Automação + volume médio
④ Azure AI Search (RAG) 🏆 ⭐⭐⭐⭐⭐ Azure metered Sem limite Enterprise, milhares de docs

🏆 Arquitetura Recomendada: RAG com Azure AI Foundry

Para agentes que precisam consultar documentos grandes, PDFs escaneados, tabelas complexas e um volume crescente de conteúdo, a arquitetura ideal combina os componentes do Azure AI Foundry no padrão RAG (Retrieval-Augmented Generation).

Orquestração de Multi-Agentes (Connected Agents)

Por que Orquestrar Múltiplos Agentes?

À medida que a adoção evolui, surge uma armadilha comum: o "agente monolítico" que tenta resolver tudo (RH + Finanças + Operações + TI) com dezenas de tópicos, conectores e instruções. O resultado é previsível — respostas inconsistentes, dificuldade de manutenção, governança confusa e custo crescente. A solução é decompor em agentes especialistas coordenados por um agente orquestrador.

✅ Benefícios da Arquitetura Multi-Agente
  • Especialização — cada agente domina um domínio (RH, Finanças, Jurídico) com instruções e knowledge focados
  • Governança distribuída — cada área é dona do seu agente, mantém seu próprio knowledge e ciclo de release
  • Reuso — um agente especialista pode ser invocado por múltiplos orquestradores
  • Manutenibilidade — alterar um especialista não impacta os demais
  • Performance — contexto menor por agente = respostas mais rápidas e precisas

Connected Agents — O Que É

Em maio/2025 o Copilot Studio liberou o recurso Connected Agents (Agent-to-Agent), que permite a um agente delegar uma tarefa a outro agente — interno ou externo — em tempo de execução. O agente "pai" funciona como roteador: interpreta a intenção do usuário e despacha para o especialista certo, recebendo a resposta de volta para compor a interação final.

Como Configurar Connected Agents

1 Criar os Agentes Especialistas Primeiro
Construa cada especialista isoladamente, com sua descrição, instruções, knowledge e topics próprios. Teste em produção com um piloto antes de conectá-lo a um orquestrador.
2 Criar o Agente Orquestrador
Novo agente no Copilot Studio com instruções de "roteador": "Você é o ponto de entrada do colaborador da empresa. Identifique a intenção e delegue ao agente especialista apropriado. Não responda perguntas técnicas diretamente — sempre encaminhe."
3 Adicionar Connected Agents
No orquestrador, em Agents → Add → Connected agent. Selecione um agente do mesmo ambiente (ou de outro ambiente compartilhado). Defina nome de exibição e, principalmente, a descrição — o orquestrador usa essa descrição para decidir quando chamá-lo.
4 Definir Inputs e Outputs
Para cada agente conectado, defina os parâmetros de entrada que o orquestrador deve passar (ex: pergunta do usuário, ID do colaborador, contexto) e os outputs esperados de retorno.
5 Testar a Roteamento
No painel de teste, faça perguntas de domínios diferentes e valide que o orquestrador está delegando corretamente. Ajuste descrições dos especialistas se houver confusão de roteamento — descrição clara é a chave.
6 Publicar Apenas o Orquestrador
Os agentes especialistas não precisam estar publicados em canais — basta estarem disponíveis no ambiente. O usuário final só conhece o orquestrador.

Boas Práticas de Design

Princípio Recomendação
Limite de Especialistas por Orquestrador Comece com 3–5 agentes conectados. Acima disso, a precisão de roteamento cai e o consumo de créditos sobe. Para mais domínios, considere orquestradores hierárquicos (orquestrador → sub-orquestrador → especialistas).
Descrições Discriminantes Cada agente conectado precisa de uma descrição que diferencie claramente seu domínio dos demais. Evite sobreposição — "responde dúvidas de RH" e "responde sobre benefícios" vão competir.
Ownership Distribuído Cada agente especialista deve ter um maker owner da área de negócio (RH owner para Agente RH, etc.). O time central de IA mantém apenas o orquestrador.
Versionamento Independente Cada especialista evolui no seu próprio ciclo (Dev → Prod via Managed Solutions). Mudanças no especialista são transparentes ao orquestrador, desde que a interface (inputs/outputs) seja preservada.
Observabilidade Habilite Application Insights em cada agente. Use o dashboard do Copilot Studio Analytics para identificar roteamento incorreto, agentes pouco usados e gargalos de latência.
Fallback Explícito Configure no orquestrador um topic de fallback para quando nenhum especialista for adequado — encaminhar para humano (Teams handoff) ou abrir ticket no ServiceNow.

Impacto no Consumo de Copilot Credits

⚠️ Atenção ao Custo

Cada chamada de agente conectado conta como uma interação adicional e consome Copilot Credits separadamente. Uma pergunta do usuário que aciona o orquestrador + 1 especialista = 2 interações faturadas. Em padrões fan-out (orquestrador + 3 paralelos) = 4 interações. Dimensione o PAYG considerando esse multiplicador e monitore o consumo médio por conversa nas primeiras semanas.

Quando NÃO Usar Multi-Agentes

  • Volume baixo (<100 interações/mês) — overhead operacional não compensa
  • Equipe sem maturidade em governança de agentes — comece com 1, evolua depois
  • Quando a sobreposição de domínios é alta — refatore os domínios primeiro

Arquitetura Mesh (Multi-Agentes Peer-to-Peer)

Por que Considerar uma Arquitetura Mesh?

O modelo Hub-and-Spoke (Seção 10) é excelente para roteamento de intenção do usuário, mas tem um limite estrutural: todo o tráfego passa pelo orquestrador. Quando os processos do negócio exigem que especialistas colaborem entre si — não só respondam ao usuário — o orquestrador vira gargalo (de latência, de governança e de manutenção). A arquitetura Mesh resolve isso permitindo que qualquer agente chame qualquer outro agente, formando uma rede peer-to-peer de especialistas.

✅ Benefícios da Arquitetura Mesh
  • Colaboração nativa — agentes consultam pares para compor respostas mais ricas (ex: Compras consulta Jurídico durante uma cotação)
  • Workflows cross-funcionais — processos ponta a ponta sem reescrever o orquestrador a cada nova integração
  • Sem ponto único de falha — não há "agente rei"; falha de um especialista não derruba toda a experiência
  • Evolução orgânica — novas conexões surgem conforme os domínios maduram, sem refatoração central
  • Reuso máximo — cada agente é simultaneamente provedor e consumidor de capacidades

Mesh — O Que É Tecnicamente

No Copilot Studio, a arquitetura Mesh é implementada usando o mesmo recurso de Connected Agents (e/ou agent-to-agent via MCP/Foundry), porém sem hierarquia: cada agente declara, em sua configuração, quais pares ele pode invocar. Não existe um agente "pai" — existem múltiplos agentes que se conhecem e se chamam conforme a necessidade. O usuário final pode entrar pela porta que fizer mais sentido (Teams, SharePoint, app de campo) e o agente acessado consulta os demais sob demanda.

Mesh vs. Hub-and-Spoke — Quando Escolher Cada Um

Critério Hub-and-Spoke (Seção 10) Mesh (Seção 11)
Ponto de entrada Único (orquestrador) Múltiplos (qualquer agente especialista)
Roteamento Centralizado no orquestrador Distribuído — cada agente conhece seus pares relevantes
Caso de uso ideal "Assistente Corporativo" único para o colaborador Workflows cross-funcionais (P2P, O2C, RH+TI, etc.)
Governança Time central mantém o orquestrador Distribuída — exige contratos de interface (inputs/outputs) bem definidos
Latência Maior (sempre passa pelo hub) Menor em chamadas pontuais; pode crescer em cadeias longas
Maturidade necessária Baixa — é o ponto de partida natural Alta — requer observabilidade madura e versionamento disciplinado
Custo (Copilot Credits) Previsível: 1 hub + 1 spoke = 2 interações Variável: cadeias de 3-4 agentes podem multiplicar o consumo
💡 Padrão Híbrido — Recomendado para Maioria dos Casos

Na prática, raramente se adota Mesh "puro". O padrão mais maduro é híbrido: um orquestrador (Hub-and-Spoke) atende o usuário final, e os especialistas adicionalmente conversam entre si (Mesh) para resolver workflows complexos. Você ganha o melhor dos dois mundos: simplicidade na entrada do usuário + colaboração rica no back-end.

Como Configurar uma Rede Mesh

1 Mapear os Workflows Cross-Funcionais Reais
Antes de configurar conexões, desenhe os 3-5 processos que cruzam departamentos hoje (ex: onboarding, aprovação de compra, análise de risco contratual). Cada workflow vai revelar uma "aresta" entre dois agentes — essas são as conexões mesh que valem a pena criar.
2 Definir Contratos de Interface por Agente
Cada agente especialista deve documentar publicamente, no seu próprio repositório de governança, quais capacidades expõe (tópicos invocáveis), inputs esperados, outputs garantidos e SLA de resposta. Trate como um contrato de API — mudanças quebram consumidores.
3 Criar Conexões Bidirecionais com Connected Agents
Em cada agente, vá em Agents → Add → Connected agent e adicione apenas os pares que ele realmente precisa invocar (não conecte tudo com tudo — explosão combinatória mata governança). Use descrições discriminantes que deixem claro quando chamar cada par.
4 Estabelecer um Catálogo Central de Agentes
Mantenha um registro vivo (SharePoint, Power Pages ou CoE Kit) listando todos os agentes da rede, seus owners, capacidades expostas, dependências e versão atual. Este catálogo é a "service map" da sua arquitetura mesh — sem ele, ninguém sabe o que existe.
5 Implementar Loop Detection e Profundidade Máxima
Em mesh, o agente A pode chamar B que chama C que chama A — loop. Configure no orquestrador (ou via Power Automate intermediário) um limite de profundidade de chamadas (ex: máximo 3 saltos) e logging de cadeia de invocação para detectar ciclos.
6 Publicar nos Canais de Usuário
Diferente do Hub-and-Spoke, em mesh vários agentes podem ser publicados diretamente no Teams/SharePoint para suas audiências naturais (RH para colaboradores, Compras para área de suprimentos). A colaboração entre eles fica invisível para o usuário final.

Boas Práticas de Design Mesh

Princípio Recomendação
Conexões Mínimas Necessárias Não conecte todos com todos. A regra é: uma conexão = um workflow real e validado. Mesh com 6 agentes deveria ter 8-12 arestas, não 30.
Idempotência nas Chamadas Agentes invocados como pares devem retornar o mesmo resultado para a mesma entrada. Operações com efeitos colaterais (ex: criar registro) devem usar IDs de transação para evitar duplicação em cenários de retry.
Versionamento Semântico de Interfaces Quando um agente muda inputs/outputs, publique como nova versão (v2) e mantenha a v1 ativa por um período de migração. Avise consumidores via catálogo central.
Observabilidade End-to-End Habilite Application Insights em todos os agentes da rede e propague um correlation ID em cada chamada. Sem isso, é impossível depurar uma cadeia de 4 agentes que falhou no terceiro salto.
Circuit Breaker Se um agente par está fora do ar ou lento, configure fallbacks locais (resposta degradada baseada em cache ou knowledge próprio) em vez de propagar a falha ao usuário.
Governança Federada Cada agente tem um maker owner; mas a arquitetura mesh como um todo precisa de um "arquiteto de plataforma" (CoE) que aprova novas conexões, revisa contratos e mantém o catálogo.
Segurança e Permissões Quando o agente A chama o agente B, decida explicitamente se B usa a identidade do usuário original (delegação) ou uma identidade de serviço (managed identity). Documente — isso impacta DLP e audit trail.

Impacto no Consumo de Copilot Credits

⚠️ Mesh Pode Multiplicar o Custo Rapidamente

Em mesh, uma única pergunta do usuário pode disparar uma cadeia de invocações: Compras → Jurídico → RH → resposta consolidada = 4 interações faturadas. Cenários complexos podem chegar a 6-8 saltos. Recomendações:

  • Defina um orçamento por workflow (ex: "aprovação de compra não pode consumir mais que 5 interações")
  • Use a calculadora de Copilot Credits (Seção 4) para estimar consumo médio antes do go-live
  • Monitore semanalmente o "chain depth" médio nas primeiras 8 semanas — se subir descontroladamente, refatore conexões
  • Considere caching de respostas de agentes pares (Power Automate + Dataverse) para perguntas repetitivas

Roadmap Sugerido de Adoção Mesh

Fase Escopo Entregável
Fase 1 — Pré-requisito Ter pelo menos 4-5 agentes especialistas maduros já em produção (saída natural da Seção 10) Agentes estáveis, com ownership definido e knowledge consolidado
Fase 2 — Catálogo e Contratos Documentar capacidades, inputs/outputs e SLAs de cada agente em um registro central Catálogo publicado (SharePoint ou Power Pages); contratos versionados
Fase 3 — Primeiras Conexões P2P Implementar 2-3 conexões mesh para os workflows cross-funcionais de maior valor (ex: Compras ↔ Jurídico) Workflows cross-funcionais funcionando; correlation ID propagado; dashboards de chain depth
Fase 4 — Expansão Controlada Adicionar novas arestas conforme novos workflows emergem; manter governança de "uma conexão = um workflow validado" Rede mesh com 8-12 conexões mapeadas; reuniões mensais de arquitetura
Fase 5 — Otimização Contínua Refatorar conexões pouco usadas, consolidar agentes redundantes, introduzir caching e circuit breakers Custo por conversa otimizado; SLA de 99% nas chamadas peer-to-peer

Quando NÃO Usar Arquitetura Mesh

  • Adoção inicial — comece sempre com Hub-and-Spoke (Seção 10); mesh é evolução, não ponto de partida
  • Sem CoE ou arquiteto de plataforma — mesh sem governança vira "spaghetti architecture" em poucos meses
  • Workflows não foram validados — não crie conexões "por via das dúvidas"; cada aresta tem custo de manutenção
  • Observabilidade imatura — sem Application Insights e correlation IDs, depurar mesh é praticamente impossível
  • Domínios fortemente acoplados — se RH e Financeiro precisam estar sempre juntos, talvez sejam um único agente

Perguntas Frequentes

1. Posso criar um agente de graça?

Sim. O Copilot Chat (gratuito) permite criar agentes simples com instruções e arquivos enviados. Qualquer usuário com conta Microsoft corporativa pode acessar copilot.microsoft.com e criar agentes pessoais sem custo adicional. Porém, esses agentes não acessam dados corporativos (emails, SharePoint, Teams) e não podem executar ações automatizadas.

2. Meu agente do M365 Copilot pode ser usado por quem não tem licença?

Depende. Se o agente for publicado apenas para usuários licenciados do M365 Copilot (via Teams, por exemplo), está incluso na licença. Se quiser disponibilizar para todos os colaboradores (inclusive sem licença Copilot), o agente precisa ser publicado via Copilot Studio e passará a consumir Copilot Credits por interação.

3. Qual a diferença prática entre um agente do Copilot Chat e um do M365 Copilot?

O agente do Copilot Chat é um assistente pessoal — apenas o criador (ou quem tem o link) consegue usá-lo, e ele só conhece os arquivos que você fornece. O agente do M365 Copilot tem acesso ao Microsoft Graph da organização, podendo consultar emails, arquivos do SharePoint/OneDrive, chats do Teams e calendário do usuário. Isso torna o agente do M365 Copilot muito mais poderoso para cenários corporativos, mas exige licenciamento.

4. Quando devo usar o Copilot Studio em vez do M365 Copilot?

Use o Copilot Studio quando precisar de:
  • Automação de processos — executar ações (criar ticket, atualizar CRM, enviar email automatizado)
  • Canais externos — website, WhatsApp, app mobile (usuários sem M365)
  • Base de conhecimento avançada — Azure AI Search, Dataverse, APIs custom
  • Governança enterprise — DLP, ambientes segregados, ALM, compliance
  • Integração com sistemas legados — via conectores Power Platform ou HTTP
  • Volume alto — centenas ou milhares de usuários interagindo simultaneamente

5. Os agentes do M365 Copilot consomem créditos do Copilot Studio?

Não automaticamente. Agentes declarativos no M365 Copilot que usam apenas knowledge (fontes de dados) estão inclusos na licença M365 Copilot do usuário. Porém, se o agente executa ações via conectores, utiliza tópicos customizados ou é acessado por usuários sem licença, o consumo passa a ser medido em Copilot Credits — que precisam ser adquiridos via Copilot Studio (PAYG ou pacote).

6. Posso compartilhar meu agente com toda a organização?

Plataforma Como compartilhar Quem pode usar
Copilot Chat Link compartilhável Apenas quem tem Copilot Chat (gratuito ou pago)
M365 Copilot Teams App Catalog (admin deve aprovar) Usuários com licença M365 Copilot
Copilot Studio Teams, SharePoint, Website, Mobile, WhatsApp, etc. Qualquer pessoa — não precisa de licença

7. Qual o custo de cada tipo de agente?

Tipo Custo para criar Custo por interação Custo para o usuário final
Copilot Chat Gratuito Incluso Precisa de Copilot Chat (gratuito)
M365 Copilot Licença M365 Copilot Incluso* Precisa de licença M365 Copilot
Copilot Studio Licença ambiente Power Platform Copilot Credits Gratuito — sem licença necessária

*Enquanto usar apenas knowledge sources. Ações com conectores consomem Copilot Credits.

8. O que são Copilot Credits e como funcionam?

A partir de 2025, a Microsoft unificou a medição de consumo do Copilot Studio em Copilot Credits. Cada tipo de interação consome uma quantidade diferente de créditos:

Tipo de Interação Consumo Aproximado
Resposta clássica (tópico pré-definido) ~0,2 créditos
Resposta generativa com IA (RAG) ~1-2 créditos
Ação via conector / Power Automate ~1 crédito por execução
Chamada a Azure OpenAI / AI Builder Variável (depende do modelo)

Os créditos podem ser adquiridos via PAYG (paga pelo consumido via Azure) ou pacotes pré-pagos (25.000 créditos/mês incluso no plano Copilot Studio). Consulte a documentação oficial de licenciamento para valores atualizados.

9. Posso começar simples e evoluir?

✅ Jornada Recomendada

A abordagem mais eficaz é progressiva:

  1. Fase 1 — Experimentar: Criar agentes no Copilot Chat (gratuito) para validar casos de uso e engajar stakeholders
  2. Fase 2 — Corporativo: Usar M365 Copilot para agentes que precisam de dados corporativos (emails, SharePoint, Teams)
  3. Fase 3 — Enterprise: Migrar para Copilot Studio quando precisar de automação, governança avançada ou canais externos

Agentes criados no M365 Copilot podem ser migrados para o Copilot Studio quando a complexidade exigir. Não é necessário recomeçar do zero.
